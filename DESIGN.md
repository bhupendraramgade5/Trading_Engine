# Matching Engine Design Document
## Trading Engine — Design Document

## 1. Overview

This project implements a multi-symbol trading engine consisting of:

*   Matching Engine
*   Order Book
*   Position Engine
*   Command Parser

The system simulates a simplified exchange supporting:

*   Limit order matching
*   Multi-symbol trading
*   Position tracking
*   Realized and Unrealized PnL
*   Command-based interface

## 2. High-Level Architecture

        CommandParser
            ↓
        MatchingEngine
            ↓
        OrderBook (per symbol)
            ↓
        Trade Callback
            ↓
        PositionEngine

### Component Responsibilities

| Component      | Responsibility                |
| -------------- | ----------------------------- |
| CommandParser  | Parse input commands          |
| MatchingEngine | Route commands & manage books |
| OrderBook      | Match orders                  |
| PositionEngine | Track positions & PnL         |


## 3. Core Design Decisions
### 3.1 Match Before Insert

Incoming orders are matched before inserting into book.

    Example:

    Sell 100 qty @ 100

    Incoming:
    Buy 150 qty @ 100

    Result:

    Match 100
    Insert remaining 50

This follows real exchange behavior:

Incoming Order → Aggressive

Existing Orders → Passive

### 3.2 Price-Time Priority

Orders are matched using:

Best price

Earliest order
```C
Implementation:
    std::map<Price, std::deque<Order>>

    Buy Side
        std::greater<>
        Highest price first.

    Sell Side
        std::less<> 
        // By default the Priority is Set TO this 
        Lowest price first.
```

## 4. Matching Engine Design
### Responsibilities

#### Matching Engine:

* Create OrderBook per symbol
* Route commands
* Generate order IDs
* Handle trade callbacks
* Connect Position Engine
* OrderBook Creation

Lazy initialization:

`getOrderBook(symbol)`

If book does not exist → create one.

This allows **multi-symbol support.**

## 5. Order ID Generation
Order IDs are generated by Matching Engine.

Reason:

* Prevent collisions
* Avoid user errors
* Ensure uniqueness

Implementation: `OrderId nextOrderId()`

## 6. Command Design

Commands are implemented using:
```C++
std::variant
Example:

    using Command =
            std::variant<
                NewOrderCommand,
                CancelOrderCommand,
                ModifyOrderCommand,
                PrintCommand,
                PriceCommand,
                PositionCommand
            >;

```
Dispatching: `std::visit()`

This removes:

    if/else chains
    virtual inheritance

## 7. OrderBook Design
**Data Structures**

``` C
Buy Side  -> map<Price, deque<Order>>
Sell Side -> map<Price, deque<Order>>
```

**Benefits**
* Price priority
* Time priority
* Fast matching
## 9. Order Lookup

For O(1) cancel & modify:

`unordered_map<OrderId, Location>`

**Location stores:**
* side
* price
* iterator

## 10. Trade Callback Design
OrderBook emits trade:

        TradeCallback

MatchingEngine receives:

        onTrade()

PositionEngine updates:

        onFill()

This creates loose coupling.

## 11. Position Engine Design

**Tracks:**

* Net Quantity
* Average Price
* Realized PnL
* Unrealized PnL
* Realized PnL

**When position closes:**

        (realized)
**Unrealized PnL :**

        (lastPrice - avgPrice) * qty
